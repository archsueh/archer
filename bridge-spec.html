Architecture Note

Agent Bridge

Cross-pane agent-to-agent orchestration for Archer

基于 ShawnPana/smux(MIT)的模型 · 适配 Archer / libghostty · v1 草案

1 · 核心思想

smux 不是一个独立 app,而是在 tmux 之上加一层寻址。每个 agent(Claude Code / Codex / Hermes…)就是跑在一个 pane 里的普通终端进程;smux 不接管进程,只提供一个 CLI 把"读屏 / 发键"包装成带 label 的命令,让任意能跑 bash 的 agent 都能操作隔壁 pane。

没有后端、没有数据库、没有消息总线。协作 = 一个 agent 把另一个 pane 当成可读写的终端。

2 · smux 的三个零件

  1. tmux.conf — 给人用的键位(Option 无前缀、鼠标、pane 标签、状态栏),全部落在 ~/.smux/。
  2. tmux-bridge CLI — 唯一的"胶水"。把 tmux 两个原语 capture-pane(读)和 send-keys(写)包成带 label 寻址的命令。
  3. smux skill — 一份 SKILL.md,教 agent "要找隔壁 agent 就调 tmux-bridge"。
# tmux-bridge 内部就是这两条 tmux 原语
read  codex 20   → tmux capture-pane -t codex -p | tail -20
type  codex "review src/auth.ts"   → tmux send-keys -t codex "…"
keys  codex Enter   → tmux send-keys -t codex Enter

3 · agent 之间怎么对话

没有协议握手。Claude Code 想让 Codex 干活,就是它自己在 shell 里执行 tmux-bridge type codex "…" + keys codex Enter,再 read codex 把屏幕读回当结果。一次协作 = 三步:type → keys → read。

4 · 映射到 Archer

Archer 的 pane 系统 = tmux 的 pane;`bridge.html` = tmux-bridge 的可视化层。真正落地只需在 Swift / libghostty 侧暴露等价能力:

smux / tmux
Archer 实现
capture-pane -t %id
读 libghostty pane 的屏幕缓冲
send-keys -t %id
向 pane 注入 input(已有输入管线)
pane label (name)
PaneRegistry: @label → SurfaceID
tmux-bridge CLI

Archer skill:教 agent 调 read/type/keys

5 · 落地步骤

  1. PaneRegistry — 给每个 surface 一个稳定 @label(claude / codex / hermes),维护 label → SurfaceID 映射,UI 上可重命名。
  2. 三个原语 — bridge.read(label, n) / bridge.type(label, text) / bridge.keys(label, key…),直接走 libghostty 的缓冲读取与输入注入。
  3. 暴露给 agent — 包成 CLI(`archer-bridge …`)P0 或 MCP server P1,让 agent 在自己的 shell/工具里调用。
  4. skill — 写一份 Archer skill 描述这三条命令与 @label 约定,装进 ~/.claude·~/.codex,agent 即学会跨 pane 协作。
  5. 可视化 — `bridge.html` 作为 Bridge Log 面板接真实事件流(read/type/keys/reply),驾驶舱可从右键 pane 打开。

6 · MCO — 并行共识补充层

Bridge 是串行交接（A → B → A），MCO 是并行扇出 + 共识（A → [B‖C‖D] → 最优解回 A）。两者互补，不替换。

项目状态：mco-org/mco 作者已转向 Hive 项目，MCO 本体可用但无主动维护。以下映射基于 MCO v0.x CLI + MCP server 模式。

MCO 能力
Archer 映射 / 价值
mco run --agents A,B,C "prompt"
同时向多个 pane 注入同一任务，Bridge 无需串行转发
并行扇出 + 等待
Cockpit 多 pane 模式天然承载：每个 pane 是一个 agent 实例
consensus scoring（投票）
读取各 pane 输出，Bridge.read 轮询 → 汇总给 Claude 做裁决
mco serve（MCP server 模式）
ArcherBridge MCP P1：agent 通过工具调用而非 CLI 发起多 pane 协作
mco_doctor / mco_review
可包装为 Archer skill：claude/codex 直接调 mco 诊断/评审工具

推荐用法：bridge.type 串行交接用于 A→B 单步委托；当需要"三个 agent 同时做，取最好结果"时，改用 mco run + bridge.read 轮询收集。Bridge Log 面板应同时记录两类事件。

7 · 安全约束

注入是高权限操作。type/keys 等于替用户敲键 —— 必须:① 仅限当前 workspace 内的 pane;② 危险按键(C-c、rm 回车)默认需确认或加白名单;③ Bridge Log 全量留痕,可审计"谁对谁发了什么"。
